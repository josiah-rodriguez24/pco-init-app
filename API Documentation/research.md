## This File contains a research document on our stack with recommendations on how to operate securely and efficiently, as well as how we can build out the app to function well. use it as context when planning out builds.


Building a church dashboard and smart scheduler on Planning Center
Bottom line
You are on the right track, but only if you make one architectural shift early: Planning Center should be your system of record, not the live database that powers every screen and every scheduling calculation. Planning Center’s API is user-scoped, Personal Access Tokens cannot be used from browser-side requests, the default API limit is 100 requests per 20 seconds per authenticated user, responses follow JSON:API, webhooks are explicitly positioned as a faster and more efficient alternative to polling, and production apps should pin an API version instead of using LATEST. The practical implication is that your app should ingest and normalize Planning Center data into your own Postgres database, then serve dashboards and scheduling logic from that local model. 

There is also an important product question hiding inside your technical plan: Planning Center Services already does a surprising amount of scheduling work natively. It can auto-schedule based on last scheduled date, blockouts, conflicts, scheduling preferences, and household settings; it can schedule multiple plans at once in matrix view; and it can automatically reschedule declines to the “next best person” based on availability and previous scheduling history. That means a custom app is most justified when you want better dashboards, cross-team fairness rules, longer-horizon planning, recommendation/approval workflows, or a smarter scheduling model than Services currently offers—not when you are only recreating existing Planning Center screens. 

Recommended architecture
A strong default architecture for this app is Next.js App Router + TypeScript + Tailwind CSS on the frontend and server-rendered layer, backed by Postgres for your local data model and a small background-job system for sync and scheduling. Next.js route handlers are a good fit for your webhook endpoints, admin actions, and write-back endpoints, but Next.js itself describes this pattern as a backend-for-frontend, not a total backend replacement; route handlers are public HTTP endpoints, and they must be secured accordingly. For dashboard pages, prefer Server Components that read directly from your database instead of routing every internal read through one giant “API endpoint.” Next.js fetches are memoized within the component tree, but they are not cached by default, so you should deliberately choose where to cache and where to stream fresh data. Static export is not the right deployment target for this project, because it removes the runtime server and only supports static GET route handlers. 

For the UI layer, your current choices are sound. Tailwind remains an excellent fit for an internal dashboard, especially when paired with shadcn/ui, which is intentionally “open code” rather than a sealed component package. That gives you accessible building blocks you can own and customize. For dense admin grids, TanStack Table is especially well suited because it is headless and gives you full control over sorting, filtering, selection, and layout. For charts, Recharts is still one of the most straightforward React-native options for dashboards. 

For data access, I would keep the database layer boring and durable. Postgres is the best fit here because you need relational joins, historical scheduling data, fairness calculations, and auditability. If you prefer a SQL-first, low-magic approach, Drizzle ORM is a very good fit; its docs emphasize TypeScript schemas, SQL-like access, and lightweight design. If you want a more batteries-included ORM with generated client types, migrations, and a visual data editor, Prisma is a strong alternative. Either one can work, but Drizzle has a slight edge if you expect to write nontrivial scheduling queries and want to stay close to SQL, while Prisma has a slight edge if you want a more guided developer experience. 

For auth, the cheapest sensible default is Auth.js rather than a paid auth platform. Next.js recommends using an auth library rather than rolling your own, and Auth.js is the official Next.js integration path for the Auth.js ecosystem. For an internal church app, that usually means Google/Microsoft/email sign-in plus local role-based access in your own database. 

For background jobs, start simple and cheap: use a Postgres-backed queue instead of adding Redis on day one. Libraries such as pg-boss and Graphile Worker are specifically built to store jobs in PostgreSQL and run them in Node.js, which is a strong cost-saving move when you are already paying for Postgres. That is a very good fit for sync jobs, nightly schedule generation, notification fan-out, or recalculating fairness scores. 

Planning Center integration strategy
Because every Planning Center API request is made on behalf of a user, I would create a dedicated integration user inside your church’s Planning Center account and grant it only the permissions your app truly needs. For a single-church internal tool, Planning Center explicitly says a Personal Access Token is acceptable when you run the integration on a server you control. What you should not do is expose that PAT in client code or through NEXT_PUBLIC_ environment variables; Next.js exposes NEXT_PUBLIC_ values to the browser bundle, and Planning Center explicitly says PATs cannot be used in client-side web requests. 

Your sync layer should be designed around initial backfill + ongoing webhook updates + periodic reconciliation. Start by pulling the core entities you need—people, service types, teams, positions, plans, assigned times, plan people, schedules, blockouts, and scheduling preferences—into local tables. Then subscribe to Planning Center webhooks so changes arrive in near real time, and run a periodic replay/reconcile job to catch anything you missed. Planning Center’s webhook docs say webhooks are usually preferable to polling, that webhook payloads include a unique delivery ID/type/attempt number, and that webhooks currently have no rate limits. Meanwhile, the API docs emphasize that rate limits can change dynamically and should be respected via response headers, not hard-coded. 

Be selective about what you mirror. The People API’s sample object includes fields such as medical_notes and passed_background_check, which means your source system may contain genuinely sensitive personal data. For a scheduling/dashboard app, that is a strong reason to sync only the fields you actually need and to treat your local database like production PII from day one. 

I would also pin your Planning Center version from the beginning with X-PCO-API-Version. Planning Center supports per-app and per-request versioning, and its docs specifically warn that LATEST is not recommended for production because it is equivalent to always asking for tomorrow’s API surface. That is an easy way to avoid waking up to a broken integration because of a silent upstream change. 

If your scheduler eventually needs to write approved schedules back into Planning Center, that path exists. The Services API exposes plan_people / team_members creation and update endpoints, and the scheduling-related resources include relationships for person, plan, team, plan times, and time preference options. My recommendation would be to make your app recommend first, write second: let an admin approve the generated schedule, then write the approved assignments back to Services. 

One more Planning Center-specific insight: if you ever envision the app living inside Planning Center’s UI, Planning Center Add-ons are React-based UI components embedded in their web products. But the add-on program is clearly aimed at broader partner integrations: it starts with contacting Planning Center, goes through agreements, kickoff, beta, and review. For a one-church internal tool, a standalone app is much simpler and more realistic as a first release. 

Smart scheduler strategy
The best way to avoid coding yourself into a corner is not to start with a “magic AI scheduler.” Start with a transparent, explainable rules-and-scoring engine. Planning Center’s own scheduling model gives you a very good baseline: last scheduled date, blockouts, conflicts, scheduling preferences, household settings, and time assignments are all first-class concepts in Services. That means your first scheduler can be modeled as: hard constraints eliminate ineligible candidates; soft constraints score the eligible ones; the admin sees the recommendation and the reasons. 

A strong first version would use hard constraints such as: role qualification, team membership, no time overlap, no blockouts, max service frequency, and any role-specific guardrails for band vs. non-band teams. Soft constraints would include fairness, “time since last scheduled,” household preferences, preferred weekends, and balancing people across months. The reason I recommend this path first is that it matches the data model you already have, it is easy to test, and it produces explainable output, which matters a lot when church staff override a recommendation and want to understand why the system chose someone. The existing Planning Center auto-scheduler already prioritizes “who hasn’t been scheduled for the longest time,” while also taking blockouts, conflicts, and preferences into account; that is a proven starting heuristic for your own logic. 

Only move to a formal optimization solver when the problem actually becomes combinatorial: for example, when you want to optimize across multiple teams, multiple service types, multi-month fairness, household coordination, and skill coverage simultaneously. That is where OR-Tools CP-SAT becomes compelling. Google’s OR-Tools docs explicitly position CP-SAT for planning and scheduling, and their employee-scheduling examples show how to encode “one worker per shift,” “max one shift per day,” and fairness constraints. My advice is to keep that solver behind a narrow interface and, if you adopt it, run it as a separate batch/service layer rather than burying optimization logic deep inside your Next.js request handlers. 

That leads to a clean maturity path. Phase one is a rule-based recommendation engine in TypeScript. Phase two adds a job queue so schedule generation runs off-request. Phase three, only if needed, introduces a solver service—likely Python-backed OR-Tools—while keeping the rest of the product unchanged. That progression gives you real value early without forcing an academic optimization stack before you know your church’s actual rules.

Testing and developer workflow
Your testing strategy should mirror the architecture. For unit tests, use Vitest + React Testing Library for scoring functions, utility code, React components, and validation logic. The Next.js Vitest guide explicitly notes that async Server Components are still a weak spot for unit testing, and recommends E2E tests for those. For end-to-end tests, use Playwright and run it against a production build (next build + next start), because the Next.js Playwright guide recommends testing against production code to better match real behavior. For API mocking and contract isolation, use MSW, which is designed as a reusable mocking layer for both browser and Node.js environments. 

Concretely, I would test four things relentlessly: webhook ingestion, sync reconciliation, scheduler scoring, and write-back approval. Webhook ingestion should verify idempotency and out-of-order delivery tolerance. Sync reconciliation should prove that a fresh backfill and an event-driven update lead to the same local state. Scheduler scoring should have golden test cases for common church constraints. Write-back approval should verify that only approved schedules get pushed to Planning Center. That test portfolio will catch most of the mistakes that would matter to your users.

For CI, GitHub Actions is the obvious cheap choice. GitHub’s Node.js CI docs support matrix builds across Node versions, and the dependency caching docs note that the setup actions can create and restore dependency caches with minimal configuration. For a small project, that is enough to run typecheck, lint, unit tests, and Playwright on pull requests. 

Because you are using Cursor, you can materially improve AI-assisted development by leaning into the tooling that now exists for this. Next.js ships version-matched docs inside the next package and documents an AGENTS.md workflow for AI coding agents, while Cursor supports persistent project rules and MCP integrations. Next.js 16+ also has MCP support for coding agents via next-devtools-mcp. In practice, that means you should keep a short AGENTS.md in the repo root that describes your architecture, naming conventions, auth rules, and “do not bypass the sync layer” constraints, then let Cursor and Next’s bundled docs keep suggestions current. 

If you choose Neon or Supabase, preview branches are also worth using early. Neon’s docs describe branching as isolated, copy-on-write database branches that fit development and preview workflows; Supabase likewise supports preview/staging branches tied to deployment flows. For schema-heavy work—especially scheduling logic—that is a safer workflow than hand-managing test databases. 

Deployment and the cheapest sensible route
For development and staging, the cheapest comfortable route is to stay managed: Vercel Hobby + Neon Free is hard to beat for convenience. Vercel Hobby is free, and Neon Free gives you 100 projects, 100 CU-hours per project, and 0.5 GB storage per project. That is excellent for prototyping, preview environments, and early internal testing. 

For real production use with church data, I would separate “cheapest possible” from “cheapest sensible.” The cheapest sensible low-ops route is usually Vercel Pro + Neon Launch or Supabase Pro. Vercel Pro starts at $20/month, and its function limits are materially friendlier than Hobby; Hobby cron jobs are limited to once per day and Vercel explicitly says timing is not guaranteed precisely, while function duration defaults to 5 minutes and tops out at 5 minutes on Hobby. Neon Launch is usage-based with a published typical spend of $15/month and a 7-day restore window. Supabase Pro starts at $25/month, includes daily backups with 7-day retention, and includes $10/month in compute credits. If your church will depend on this app weekly, the backup/recovery step up from the free database tiers is worth the small added cost. 

If you need an always-on worker for schedule generation and you want to keep costs low, a small Node worker on a generic host can be cheaper than forcing everything through serverless request lifecycles. Railway’s pricing docs show a $5/month Hobby plan with $5 of included usage and platform-level usage caps, which makes it appealing for a lightweight worker process. That can be a good home for pg-boss or Graphile Worker while your web app stays on Vercel. 

The absolute lowest-dollar production route is self-hosting on a small VPS. Hetzner’s cloud pricing currently shows very low entry points, including plans like CX23 from €3.49/month max in at least some regions/configurations, and Hetzner bills hourly up to the monthly cap. If you go this route, follow Next.js’s own self-hosting guidance and put nginx or another reverse proxy in front of your app rather than exposing Next directly. This route is cheapest on paper, but it shifts backups, patching, monitoring, and incident response onto you. For church data, I would only choose it if you are genuinely comfortable running Linux, Postgres backups, and TLS yourself. 

My recommendation for your specific case is straightforward: build on Vercel + Postgres first, keep the scheduler engine separate from the UI, and use Planning Center as an upstream source—not as your live query layer. Start with Next.js App Router, Tailwind, shadcn/ui, Postgres, Auth.js, and a Postgres-backed job queue. Launch the first version as a dashboard plus schedule recommendations, not a fully autonomous scheduler. Only add a solver service, a dedicated worker host, or a Planning Center embedded add-on after the church is already getting value from the simpler version. That path is the best balance of speed, cost, and avoiding architectural dead ends.
